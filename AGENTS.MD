## 1. 项目全景

| 维度       | 说明                                                                                         |
| ---------- | -------------------------------------------------------------------------------------------- |
| 项目名称   | 人工智能中文教程（Artificial Intelligence Tutorial (zh)）                                     |
| 适用对象   | 高中毕业水平、零基础起步，希望系统学习人工智能的人                                           |
| 学习目标   | 从数学、编程、数据、模型、工程到研究，建立完整能力闭环                                       |
| 当前结构   | **5 大阶段** → **18 个主题模块** → **180 个知识主题** → **807 个知识点目录**                   |
| 组织层级   | `阶段/主题模块/知识主题/知识点`                                                              |
| 命名规则   | 全部使用中文目录名，通过数字前缀保留学习顺序（`XX_名称`）                                    |
| 核心原则   | 先基础后热点 → 先原理后工具 → 先验证后结论 → 先落地后扩展                                    |
| 文件格式   | 教程内容使用 **Markdown（.md）** 文件呈现                                                    |
| 版本管理   | Git 仓库，`.gitkeep` 占位空目录                                                              |

### 学习哲学（五不原则）

1. 不从热点工具入门，而从底层能力出发。
2. 不把数学、编程、数据、模型割裂成彼此无关的学科。
3. 不把工程部署视为附加项，而把它视为专业能力的一部分。
4. 不把论文阅读和趋势判断留到最后，而是在理解原理后尽早介入。
5. 不把伦理、安全与产品思维当作附录，而把它们纳入完整的专业判断体系。

## 4. 开发规范

### 4.1 目录与文件命名

| 规则       | 示例                                    | 说明                                                        |
| ---------- | --------------------------------------- | ----------------------------------------------------------- |
| 数字前缀   | `01_回归`、`02_分类`                    | 保持学习顺序稳定                                            |
| 中文命名   | `03_概率论与统计`                       | 目录和文件统一使用中文                                      |
| 层级一致   | `阶段/主题模块/知识主题/知识点`          | 四级结构不随意增减                                          |
| 子讲拆分   | `知识点/01_第一讲.md`                   | 如需拆分，使用数字前缀子文件                                |

### 4.2 文件组织约定

每个知识点目录中应包含以下内容（以单个 `.md` 文件或多个文件形式存在）：

```
知识点目录/
├── 知识点名称.md           ← 课程主文件（必须），文件名与知识点目录同名
├── 01_第一讲.md            ← 分讲文件（按需，内容较长时可拆分）
├── 02_第二讲.md            ← 分讲文件（按需，内容较长时可拆分）
├── assets/                ← 图片、图表资源（按需）
├── code/                  ← 示例代码文件（按需）
└── exercises/             ← 练习题与参考答案（按需）
```

- **课程主文件命名**：使用与知识点目录同名的 `.md` 文件作为课程主文件，**不要使用 `README.md`**。例如知识点目录为 `01_一元二次方程/`，则课程主文件为 `01_一元二次方程/01_一元二次方程.md`。这样做的目的是方便知识点之间通过文件名进行相互引用和跳转，而非所有文件都叫 `README.md` 导致引用路径不清晰。
  > ⚠️ **注意**：这意味着在 GitHub 上浏览知识点目录时，不会自动显示课程内容（因为 GitHub 只自动渲染 `README.md`）。读者需要点击对应的 `.md` 文件来查看课程。这是为了跨知识点引用清晰性而做出的有意取舍。
- 当内容较长或需要拆分为多讲时，使用 `01_第一讲.md`、`02_第二讲.md` 等格式，同时保留主文件作为该知识点的概览和导航页。
- **知识主题概览页**：每个知识主题（第三级目录）仍使用 `README.md` 作为概览和导航页，因为概览页不需要被其他知识点直接引用。
- **跨知识点链接格式**：链接其他知识点时，应直接指向具体的 `.md` 文件；**相对路径以当前文件所在目录为准**。例如：
  - 从**知识点主文件**发起链接（当前文件位于某个知识点目录内）时，可写为：`../02_不等式与绝对值/02_不等式与绝对值.md`。
  - 从**知识主题概览页 `README.md`** 发起链接（当前文件位于知识主题目录内）时，可写为：`./02_不等式与绝对值/02_不等式与绝对值.md`。
  - 如果目标知识点的课程尚未创建（仍为空目录），则也应按同样规则、基于当前文件所在目录链接到目录路径；例如从知识点主文件发起可写为 `../02_不等式与绝对值/`，从知识主题概览页发起可写为 `./02_不等式与绝对值/`。待课程创建后再更新为具体文件链接。

### 4.3 Git 约定

- 空目录使用 `.gitkeep` 占位。
- 课程内容创建后，删除对应的 `.gitkeep` 文件。
- 提交信息使用中文，格式：`[阶段编号/主题模块] 动作：知识点名称`。
  - 示例：`[02/深度学习] 新增：01_神经网络课程内容`
- 不要提交编辑器配置、缓存或临时文件（已在 `.gitignore` 中配置）。

### 4.4 禁止事项

- 不得在课程内容中硬编码绝对路径。
- 不得引入与教程无关的第三方框架依赖（知识图谱可视化除外）。
- 不得修改 `知识图谱可视化/` 目录下的文件（该模块独立维护）。
- 不得删除尚未被课程内容替代的 `.gitkeep` 文件。
- 不得在知识点目录中使用 `README.md` 作为课程主文件名，应使用与知识点目录同名的 `.md` 文件（详见 4.2）。
- 不得在参考资料中引用版权不明确的资源（详见 4.5）。
- 不得使用子代理（sub-agent / task agent）进行 git 推送操作，只能由主代理通过 `report_progress` 工具推送代码，否则会导致推送失败。

### 4.5 参考资料引用规范

- **只引用确定开源或公开可访问的知识资源**，避免引用可能造成版权纠纷的内容。
- 推荐引用以下类型的资源：
  - 开源教材或以 Creative Commons 等开放许可发布的书籍
  - 官方文档（如 Python 官方文档、PyTorch 官方教程等）
  - 作者本人公开发布的博客、视频或教程（如 YouTube 上的公开教育频道）
  - 学术论文（arXiv 预印本或开放获取论文）
  - 维基百科等公共知识库
- **不得引用**以下类型的资源：
  - 付费教材或版权受保护的书籍（除非明确有开放获取版本）
  - 需要登录或付费才能访问的内容
  - 版权归属不清晰的网络资源
- 每条参考资料应注明其开放获取的性质（如 "开源教材"、"官方文档"、"CC BY 许可" 等）。

### 4.6 课程行文质量要求

- **充分使用引导性文字**：在每个概念、公式、代码块之前，都应有过渡性和引导性的自然语言文字，帮助读者理解"为什么要学这个"、"这和前面讲的有什么关系"。
- **段落间要有衔接**：避免知识点之间的突然跳跃，使用承上启下的句子自然过渡。例如"前面我们学会了因式分解法来解方程，但你可能会问：如果方程不容易分解怎么办？别担心，接下来我们学习一种更通用的方法——配方法。"
- **对每个核心概念进行充分解释**：不只是给出定义和公式，还要用自己的话解释它的含义、直觉、适用场景和局限性。目标是让零基础读者仅通过阅读文字就能理解概念。
- **增加理解性提问**：在关键概念讲解后，可以插入反思性问题引导读者思考，如"想一想：为什么 $a$ 不能等于 0？"
- **讲解要有层次感**：每个知识板块应遵循"提出问题 → 直觉引导 → 正式定义 → 举例验证 → 总结过渡"的节奏。

#### 5.3.2 叙事引导式教学策略

> **核心理念**：教程不是概念词典，而是一段引导读者逐步理解的旅程。每篇课程应该像一篇优秀的技术博文，有起承转合，让读者在阅读过程中自然地学会每个概念。

1. **先直觉后形式**：任何概念先用日常语言或类比解释，再给出数学/技术定义。
2. **先简单后复杂**：先展示最简单的例子，再逐步引入变化和复杂度。
3. **先具体后抽象**：先用具体数字/场景演示，再抽象为通用公式/模式。
4. **先手动后自动**：先手算/手写代码，再引入库函数/框架 API。
5. **先单一后组合**：先掌握单一概念，再讲概念组合。
6. **叙事驱动**：用连贯的行文串联所有概念，而不是将概念逐条罗列。正文应有明确的叙事线索（提出问题 → 分析问题 → 引入概念 → 验证解决 → 自然过渡到下一个概念）。
7. **就地图解**：所有图表、流程图、关系图必须插入在它所解释的概念旁边，作为正文叙事的一部分，而不是集中在单独的"图解"章节。
8. **关键词超链接**：正文中出现的知识点关键词，应通过 Markdown 超链接直接指向对应的知识点目录，方便读者按需跳转。格式示例：`**[梯度下降（Gradient Descent）](../04_最优化/02_梯度下降/)**`。注意：加粗超链接 `**[文字](URL)**` 的前后必须各留一个空格，否则在某些 Markdown 渲染器中粗体标记会显示为原始的 `**` 而不会被正确渲染。

#### 5.3.3 代码示例要求

| 要求               | 说明                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| 语言               | 默认使用 **Python**；涉及前端可用 JavaScript，涉及底层可用 C/C++                 |
| 可运行性           | 所有代码必须可以直接复制运行，不得出现伪代码占位                                 |
| 注释               | 关键步骤必须有中文注释                                                           |
| 环境声明           | 在代码块前注明依赖库及版本                                                       |
| 输出展示           | 给出预期输出，方便学习者自检                                                     |
| 复杂度控制         | 优先使用标准库和主流库（numpy, pandas, scikit-learn, torch, tensorflow）          |
| 渐进示例           | 同一知识点的多个代码示例应由简到繁                                               |

#### 5.3.4 数学公式要求

| 要求               | 说明                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| 格式               | 使用 LaTeX 语法的 `$...$`（行内）和 `$$...$$`（独立行）                          |
| 行内公式前后空格   | 行内公式 `$...$` 的前后必须各加一个空格，避免渲染异常。**包括中文标点符号之后也必须加空格**。例如写 `系数 $a$ 的值` 而非 `系数$a$的值`；写 `其中： $x$ 表示` 而非 `其中：$x$ 表示`；写 `递增。 $\square$` 而非 `递增。$\square$`；写 `变窄， $y = 0.5x^2$ 变宽` 而非 `变窄，$y = 0.5x^2$ 变宽` |
| 独立行公式前后空行 | 独立行公式 `$$...$$` 的前后必须各空一行（即 `$$` 所在行的上一行和下一行都必须是空行），否则部分 Markdown 渲染器无法正确渲染为独立公式块。单行形式 `$$公式$$` 同样需要前后各空一行 |
| 禁止连续公式紧贴   | 严禁将多个独立行公式 `$$...$$` 写在同一行或紧邻排列（如 `$$公式1$$$$公式2$$` 或 `$$公式1$$文字$$公式2$$`）。连续的 `$$` 会导致渲染器无法正确识别公式边界，产生显示错误。**每个 `$$...$$` 公式必须独占一行，与相邻公式或文字之间用空行分隔**。在 `<details>` 折叠区域中同样适用此规则：每个单行 `$$公式$$` 之间必须有空行隔开，中间的文字说明也应独立成行 |
| 符号一致           | 同一文档中同一符号的含义保持一致                                                 |
| 中文解读           | 每个关键公式后附加"直觉解读"，用自然语言解释公式含义                             |
| 推导步骤           | 省略不超过 1 步，每步标注理由                                                    |
| 符号表             | 复杂公式文档开头或公式前提供符号表                                               |

#### 5.3.5 图解要求

| 要求               | 说明                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| **就地插入**       | 图表必须紧跟在它所解释的概念之后，嵌入正文叙事中，**禁止**集中放在独立的"图解"章节 |
| 优先 Mermaid       | 流程图、关系图优先使用 Mermaid，便于版本控制                                     |
| 函数图像           | 函数图像**不要使用 Mermaid**，应使用 Python（matplotlib）生成 PNG 图片（详见下方最佳实践） |
| 备选 ASCII         | 简单结构图可使用 ASCII art                                                       |
| 图片格式           | PNG 或 SVG，放在 `assets/` 子目录中                                              |
| 图注               | 每张图必须有 `> 📌 **图解说明**：` 格式的标题和简要说明                           |
| 上下文衔接         | 图的前后各需一句话，分别说明"接下来用图来解释"和"从图中可以看到"                 |

##### 函数图像生成最佳实践

当课程中需要展示函数图像（如抛物线、正弦曲线、损失函数曲面等）时，应遵循以下规范：

**文件组织**：
- 绘图脚本放在知识点的 `code/` 子目录中，如 `code/plot_parabola.py`
- 生成的图片放在 `assets/` 子目录中，如 `assets/parabola_discriminant.png`
- 脚本应使用相对路径自动定位输出目录，确保在知识点目录下直接运行即可生成图片

**脚本编写规范**：
- 环境要求：Python 3.10+，matplotlib，numpy
- 脚本必须可独立运行（`python code/plot_xxx.py`），方便读者自行复现
- 脚本开头应注明文件名、用途和依赖库版本
- 使用 `os.path` 基于脚本自身位置计算输出路径，不硬编码绝对路径

**字体与国际化**：
- 图表中的标题、标签、图例**使用英文或 LaTeX 数学公式**，避免中文字符导致的字体缺失问题（服务器环境通常不预装中文字体）
- 中文说明放在 Markdown 正文的图注中，不放在图片内部
- 推荐字体设置：`plt.rcParams['font.sans-serif'] = ['DejaVu Sans']`
- 负号显示：`plt.rcParams['axes.unicode_minus'] = False`

**图片质量**：
- 分辨率：`dpi=150`（兼顾清晰度和文件大小）
- 背景色：`facecolor='white'`（确保在深色主题下也能正常显示）
- 保存时使用 `bbox_inches='tight'` 去除多余空白
- 使用 `plt.tight_layout()` 避免子图之间或标签与边框的重叠

**图表美化**：
- 关键数据点（如函数零点、极值点）应使用醒目颜色标注并添加注释
- 坐标轴标签使用 LaTeX 格式（如 `$x$`、`$y$`）
- 开启网格线（`alpha=0.3`）辅助读者读数
- 隐藏不必要的边框（`ax.spines['top'].set_visible(False)`）

**在正文中引用**：
```markdown
下面这张图展示了……的关系：

![图片描述](assets/文件名.png)

> 📌 **图解说明**：……你可以运行 `code/plot_xxx.py` 自行生成这张图。
```

#### 5.4.1 教学方法论

英语类课程采用 **ESP（English for Specific Purposes，专门用途英语）** 教学方法论，与通用英语教学有本质区别。核心策略包括：

| 策略 | 说明 | 在课程中的体现 |
| ---- | ---- | -------------- |
| **语块教学（Lexical Chunks）** | 将固定搭配作为整体单元教学，而非孤立单词 | 每课包含"常用语块"对照表，要求学生记忆完整搭配 |
| **词根词缀分析（Morphological Analysis）** | 利用拉丁语和希腊语词根规律批量扩展词汇 | 词汇课包含"词根词缀解码"表，训练推理能力 |
| **真实语境浸入（Authentic Context）** | 使用来自真实技术文档/论文/代码的材料 | 每课包含"真实语境阅读"段落，附逐句注释 |
| **模式识别（Pattern Recognition）** | 技术英文高度模式化，掌握固定句式即可快速理解 | 报错信息、文档、论文摘要都有固定句式模板 |
| **间隔重复（Spaced Repetition）** | 按照遗忘曲线安排复习节奏 | 每课包含间隔复习计划表 |
| **输出驱动（Output-Driven）** | 通过写笔记、制卡片、造句来巩固理解 | 总结与记笔记主题强调输出 |
| **任务驱动（Task-Based）** | 围绕真实任务（读报错、查文档）组织教学 | 每课包含真实场景的动手任务 |

## 7. 风格与排版规范
